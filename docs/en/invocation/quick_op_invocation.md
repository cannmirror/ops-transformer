# Operator Invocation

## Prerequisites

- Environment Deployment: Before invoking operators, complete the basic environment setup by referring to [Environment Deployment](../context/quick_install.md).

- Operator Invocation List: For operators that can be invoked in the project, refer to [Operator List](../op_list.md). For the aclnn interfaces corresponding to operators, refer to [aclnn List](../op_api_list.md).
- build.sh: Operator invocation depends on the build.sh script in the root directory. View its functionality through the `bash build.sh --help` command. For parameter introduction, refer to [build Parameter Description](../context/build.md).

Otherwise, refer to [Offline Compilation](#offline-compilation) to manually download third-party software.

After preparing the open-source third-party software, select the following compilation methods as needed:

- **Custom Operator Package**:

  The package generated by selecting partial operators for compilation is called a custom operator package. It acts on the CANN package in a **mounting** form and does not change the original package content. The generated custom operator package has higher priority than the original CANN package. This package supports aclnn and graph mode invocation of AI Core operators.

- **ops-transformer Package**:

  The package generated by compiling the entire project is called the ops-transformer package. It can **completely replace** the corresponding part of the CANN package. This package supports aclnn and graph mode invocation of AI Core operators.

- **ops-transformer Static Library**:

  > Note: If you need to **perform secondary publishing based on this project** and have **requirements for software package size**, it is recommended to use static library compilation. This library can be linked to your application development program, retaining only the operators needed for your business, thereby achieving minimized software deployment.

  This means the entire project is compiled into a static library file, including libcann_transformer_static.a and aclnn interface header files. This package only supports aclnn invocation of AI Core operators.

### Online Compilation

#### Custom Operator Package

1. **Compile Custom Operator Package**

    Enter the project root directory and execute the following compilation command:

    ```bash
    bash build.sh --pkg --soc=${soc_version} [--vendor_name=${vendor_name}] [--ops=${op_list}]
    # Example: Compile the FlashAttentionScore operator
    # bash build.sh --pkg --soc=ascend910b --ops=flash_attention_score
    # Compile operators in the experimental contribution directory
    # bash build.sh --pkg --experimental --soc=ascend910b --ops=${experimental_op}
    ```

    - --soc: ${soc_version} represents the NPU model. Atlas A2 series products use "ascend910b" (default), Atlas A3 series products use "ascend910_93", Ascend 950PR/Ascend 950DT products use "ascend950".
    - --vendor_name (optional): ${vendor_name} represents the name of the custom operator package being built, default name is custom.
    - --ops (optional): ${op_list} represents the operators to be compiled. When not specified, all operators are compiled by default. The format is like "apply_rotary_pos_emb,rope_quant_kvcache,...", with multiple operators separated by a comma ",".
    - --experimental (optional): Indicates compiling operators in the experimental contribution directory. ${experimental_op} is the new contributed operator directory name.

    If neither ${vendor_name} nor ${op_list} is passed, the ops-transformer package is compiled; if compiling a custom operator package for all operators, ${vendor_name} must be passed. When the following message appears, the compilation is successful.

    ```bash
    Self-extractable archive "cann-ops-transformer-${vendor_name}_linux-${arch}.run" successfully created.
    ```

    After successful compilation, the run package is stored in the build_out directory under the project root directory.

2. **Install Custom Operator Package**

    ```bash
    ./cann-ops-transformer-${vendor_name}_linux-${arch}.run
    ```

    The custom operator package installation path is `${ASCEND_HOME_PATH}/opp/vendors`. ${ASCEND_HOME_PATH} has been configured through environment variables, representing the CANN toolkit package installation path, generally ${install_path}/cann.

    > Note: If the operator package deployment specifies the installation directory through the --install-path parameter, before using the custom operator, execute the source ${install-path}/vendors/${vendor_name}/bin/set_env.bash command. The set_env.bash script appends the custom operator package installation path to the ASCEND_CUSTOM_OPP_PATH environment variable, making the custom operator effective in the current environment.

3. **(Optional) Delete Custom Operator Package**

    Note that custom operator packages do not support uninstallation. If uninstallation is needed, delete the vendors/${vendor_name} directory and delete the load_priority configuration item corresponding to ${vendor_name} in vendors/config.ini.

#### ops-transformer Package

1. **Compile ops-transformer Package**

    Enter the project root directory and execute the following compilation command:

    ```bash
    # Compile all operators except those in the experimental contribution directory
    bash build.sh --pkg --soc=${soc_version}
    # Compile all operators in the experimental contribution directory
    # bash build.sh --pkg --experimental --soc=${soc_version}
    ```

    - --soc: ${soc_version} represents the NPU model. Atlas A2 series products use "ascend910b" (default), Atlas A3 series products use "ascend910_93", Ascend 950PR/Ascend 950DT products use "ascend950".
    - --experimental (optional): Indicates compiling operators in the experimental contribution directory.

    When the following message appears, the compilation is successful.

    ```bash
    Self-extractable archive "cann-${soc_name}-ops-transformer_${cann_version}_linux-${arch}.run" successfully created.
    ```

    ${soc_name} represents the NPU model name, which is the remaining content after removing "ascend" from ${soc_version}. After successful compilation, the run package is stored in the build_out directory.

2. **Install ops-transformer Package**

    ```bash
    # Installation command
    ./cann-${soc_name}-ops-transformer_${cann_version}_linux-${arch}.run --full --install-path=${install_path}
    ```

    ${install_path}: Represents the specified installation path, which must be the same path as the toolkit package installation. The default installation is in the `/usr/local/Ascend` directory.

3. **(Optional) Uninstall ops-transformer Package**

    ```bash
    # Uninstallation command
    ./${install_path}/cann/share/info/ops_transformer/script/uninstall.sh
    ```

#### ops-transformer Static Library

> Note: The static library only supports Atlas A2 and Atlas A3 series products. Experimental operators do not support using the static library at this time.

1. **Compile ops-transformer Static Library**

    Enter the project root directory and execute the following compilation command:

    ```bash
    bash build.sh --pkg --static --soc=${soc_version}
    ```

    ${soc_version} represents the NPU model. Atlas A2 series products use "ascend910b" (default), Atlas A3 series products use "ascend910_93".
    When the following message appears, the compilation and compression are successful.

    ```bash
    [SUCCESS] Build static lib success!
    Successfully created compressed package: ${repo_path}/build_out/cann-${soc_name}-ops-transformer-static_${cann_version}_linux-${arch}.tar.gz
    ```

    ${repo_path} represents the project root directory, ${soc_name} represents the NPU model name, which is the remaining content after removing "ascend" from ${soc_version}. After successful compilation, the compressed package is stored in the build_out directory.

2. **Extract ops-transformer Static Library**

    Enter the build_out directory and execute the extraction command:

    ```bash
    tar -zxvf ./cann-${soc_name}-ops-transformer-static_${cann_version}_linux-${arch}.tar.gz -C ${static_lib_path}
    ```

    ${static_lib_path} represents the static library extraction path. The directory structure after extraction is as follows:

    ```text
    ├── cann-${soc_name}-ops-transformer-static_${cann_version}_linux-${arch}
    │   ├── lib64
    │   │   ├── libcann_transformer_static.a        # Static library file
    │   └── include
    |       ├── ...                                 # aclnn interface header files
    ```

### Offline Compilation

If compiling in an environment without internet connection, prepare the dependent third-party software in advance, then perform source code compilation. The specific process is as follows:

1. **Check Whether the Basic Environment is Complete**

    Ensure that the basic environment setup has been completed according to [Environment Deployment](../context/quick_install.md), including CANN package installation, source code download, and so on.

    - In a networked environment, go to [this project homepage](https://gitcode.com/cann/ops-transformer), and complete source code download through the `Download ZIP` or `clone` button according to the guidance.
    - Connect to the offline environment and upload the source code to your specified directory. If a source code compressed package was downloaded, extract it first.

2. **Download Third-Party Software Dependencies**
  
    Download third-party software in advance in a networked environment. Currently, the following methods are available. Select as needed:

    - Method 1: Manually download according to the table provided in [Third-Party Software Dependencies](#third-party-software-dependencies). If downloading from other addresses, ensure the version numbers are consistent.

    - Method 2: One-click download through the [third_lib_download.py](../../../scripts/tools/third_lib_download.py) script. This script is in the `scripts/tools/` directory of this project. Download the script and execute the following command:

        ```bash
        python ${scripts_dir}/third_lib_download.py
        ```

    ${scripts_dir} represents the script storage path. The downloaded third-party software packages are stored in the current script directory by default.

3. **Compile Operator Package**

   Upload the downloaded third-party software to the offline environment. It can be stored in the `third_party` directory or a custom directory. **The former is recommended, as its compilation command is consistent with the online compilation scenario command.**

    - **third_party Directory** (Recommended)
      
      Create a `third_party` directory in the project root directory (if it already exists, no need to create it), and copy the third-party software to this specified directory. At this point, the compilation command is consistent with the online compilation command. For details, refer to [Online Compilation](#online-compilation).
      
    - **Custom Directory**
      
      Create a new `${cann_3rd_lib_path}` directory anywhere in the offline environment, and copy the third-party software to this directory. Ensure this directory has access permissions.

        ```bash
      mkdir -p ${cann_3rd_lib_path}
        ```
      
      At this point, the compilation command needs to add `--cann_3rd_lib_path=${cann_3rd_lib_path}` on top of the online compilation command to specify the third-party software path. Assuming the storage path is `/path/cann_3rd_lib_path`, the commands for different compilation methods are as follows:
      
      - Custom Operator Package
      
        ```bash
        bash build.sh --pkg --soc=${soc_version} [--vendor_name=${vendor_name}] [--ops=${op_list}] --cann_3rd_lib_path=${cann_3rd_lib_path}
        # Example: Compile the FlashAttentionScore operator
        # bash build.sh --pkg --soc=ascend910b --ops=flash_attention_score --cann_3rd_lib_path=/path/cann_3rd_lib_path
        ```
        
      - ops-transformer Full Package
      
        ```bash
        bash build.sh --pkg --soc=${soc_version} --cann_3rd_lib_path=${cann_3rd_lib_path}
        # bash build.sh --pkg --soc=ascend910b --cann_3rd_lib_path=/path/cann_3rd_lib_path
        ```
        
      - ops-transformer Static Library
      
        ```bash
        bash build.sh --pkg --static --soc=${soc_version} --cann_3rd_lib_path=${cann_3rd_lib_path}
        # bash build.sh --pkg --static --soc=ascend910b --cann_3rd_lib_path=/path/cann_3rd_lib_path
        ```

4. **Install/Uninstall Operator Package**
   
    The operator package results compiled in offline and online scenarios are the same, stored in the build_out directory under the project root directory by default, and the installation and uninstallation operation commands are also the same. For details, refer to [Online Compilation](#online-compilation).

## Local Verification

Execute operators and UT test cases through the build.sh script in the project root directory. Currently, operators support API mode (aclnn interface) and graph mode invocation. **aclnn invocation is recommended**.

### Execute Operator Sample

> **Note**: Ascend 950PR products use simulation execution for operator samples. Refer to [Simulation Guide](../debug/op_debug_prof.md#method-2-for-ascend-950pr).

- Execute operator samples based on the **Custom Operator Package**. After package installation, execute the following command:

    ```bash
    bash build.sh --run_example ${op} ${mode} ${pkg_mode} [--vendor_name=${vendor_name}] [--soc=${soc_version}]
    # Example: Execute the FlashAttentionScore operator sample
    # bash build.sh --run_example flash_attention_score eager cust --vendor_name=custom
    ```

    - ${op}: Represents the operator to be executed. The operator name is in lowercase underscore form, such as flash_attention_score.
    - ${mode}: Represents the execution mode. Currently supports eager (aclnn invocation) and graph (graph mode invocation).
    - ${pkg_mode}: Represents the package mode. Currently only supports cust, which is the custom operator package.
    - ${vendor_name} (optional): Consistent with the custom operator package build setting, default name is custom.
    - ${soc_version} (optional): Represents the NPU model, default "ascend910b". When set to "ascend950", it will run the sample files in the "arch35" directory.

    Note: When ${mode} is graph, ${pkg_mode} and ${vendor_name} are not specified.

- Execute operator samples based on the **ops-transformer Package**. After installation, execute the following command:

    ```bash
    bash build.sh --run_example ${op} ${mode} [--soc=${soc_version}]
    # Example: Execute the FlashAttentionScore operator sample
    # bash build.sh --run_example flash_attention_score eager
    ```

    - ${op}: Represents the operator to be executed. The operator name is in lowercase underscore form, such as flash_attention_score.
    - ${mode}: Represents the operator execution mode. Currently supports eager (aclnn invocation) and graph (graph mode invocation).
    - ${soc_version} (optional): Represents the NPU model, default "ascend910b". When set to "ascend950", it will run the sample files in the "arch35" directory.

- Execute operator samples based on the **ops-transformer Static Library**:

    1. **Prerequisites**

        The ops-transformer static library depends on the ops-legacy static library and the ops-math static library. Prepare the above static libraries, extract them, and move all lib64 and include directories to a unified directory ${static_lib_path}.

        > Note: The ops-legacy static library `cann-${soc_name}-ops-legacy-static_${cann_version}_linux-${arch}.tar.gz` needs to be obtained by clicking the [download link](https://ascend.devcloud.huaweicloud.com/artifactory/cann-run-release/software/master). The ops-transformer static library and ops-math static library packages are not yet provided; generate them through local compilation.

    2. **Create run.sh**

        Create a run.sh file in the same directory as the operator `examples/test_aclnn_${op_name}.cpp` to be executed.

        Using the FlashAttentionScore operator executing test_aclnn_flash_attention_score.cpp as an example, the sample is as follows:

        ```bash
        # Environment variables生效
        if [ -n "$ASCEND_INSTALL_PATH" ]; then
            _ASCEND_INSTALL_PATH=$ASCEND_INSTALL_PATH
        elif [ -n "$ASCEND_HOME_PATH" ]; then
            _ASCEND_INSTALL_PATH=$ASCEND_HOME_PATH
        else
            _ASCEND_INSTALL_PATH="/usr/local/Ascend/cann"
        fi
    
        source ${_ASCEND_INSTALL_PATH}/bin/setenv.bash
    
        # Compile executable file
        g++ test_aclnn_flash_attention_score.cpp -I ${static_lib_path}/include -L ${static_lib_path}/lib64 -L ${ASCEND_HOME_PATH}/lib64 -Wl,--allow-multiple-definition \
        -Wl,--start-group -lcann_transformer_static -lcann_math_static -lcann_legacy_static -Wl,--end-group -lgraph -lmetadef \
        -lascendalog -lregister -lopp_registry -lops_base -lascendcl -ltiling_api -lplatform -ldl -lnnopbase -lgraph_base \
        -lc_sec -lunified_dlog -lruntime -lhccl_fwk -o test_aclnn_flash_attention_score   # Replace with actual operator executable file name
    
        # Execute program
        ./test_aclnn_flash_attention_score
        ```

        ${static_lib_path} represents the unified static library placement path; ${ASCEND_HOME_PATH} has been configured through environment variables, representing the CANN toolkit package installation path, generally ${install_path}/cann; replace the final executable file name with the actual operator executable file name.
        lcann_transformer_static, lcann_math_static, lcann_legacy_static represent the static library files that the operator depends on, obtained from the unified static library placement path ${static_lib_path}; lgraph, lmetadef, and so on represent the underlying library files that the operator depends on, which can be obtained from the CANN toolkit package.

    3. **Execute run.sh**

        ```bash
        bash run.sh
        ```

Regardless of the method used above, the operator sample execution will print results. Using the FlashAttentionScore operator execution as an example:

### Execute Operator UT

> Note: Executing UT test cases depends on the googletest unit testing framework. For detailed introduction, refer to the [googletest official website](https://google.github.io/googletest/advanced.html#running-a-subset-of-the-tests).

```bash
# Install test-related requirements.txt dependencies in the root directory
pip3 install -r tests/requirements.txt
# Method 1: Compile and execute UT test cases for specified operators and corresponding functions (select one)
bash build.sh -u --[opapi|ophost|opkernel] --ops=flash_attention_score
# Method 2: Compile and execute all UT test cases
# bash build.sh -u
# Method 3: Compile all UT test cases without execution
# bash build.sh -u --noexec
# Method 4: Compile and execute UT test cases for corresponding functions (select one)
# bash build.sh -u --[opapi|ophost|opkernel]
# Method 5: Compile UT test cases for corresponding functions without execution (select one)
# bash build.sh -u --noexec --[opapi|ophost|opkernel]
# Method 6: Specify soc compilation when executing UT test cases
# bash build.sh -u --[opapi|ophost|opkernel] [--soc=${soc_version}]
```

To verify whether the ophost function is normal, execute the following command:

```bash
bash build.sh -u --ophost
```

After execution, the following content appears, indicating successful execution.

```bash
Global Environment TearDown
[==========] ${n} tests from ${m} test suites ran. (${x} ms total)
[  PASSED  ] ${n} tests.
[100%] Built target transformer_op_host_ut
```

${n} represents n test cases executed, ${m} represents m test items, ${x} represents the time consumed by executing test cases in milliseconds.
